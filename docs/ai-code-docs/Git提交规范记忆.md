# Git 提交规范

## 适用范围

- 适用于当前工作区三端代码仓库的 Git 提交：
  - `backend/`
  - `admin/`
  - `minigram/`

本规范基于当前三端完整 Git 提交历史整理，目的是与仓库既有提交风格保持一致，而不是重新定义另一套风格。

---

## 历史提交风格总结

从当前完整历史提交看，项目已经形成以下稳定模式：

- 使用 **Conventional Commits** 前缀
- 提交标题以 **中文描述** 为主
- 常见类型主要是：
  - `feat`
  - `fix`
  - `refactor`
  - `chore`
- 少量历史提交出现过：
  - `style`
  - `docs`
- 标题通常是单行短句，严禁使用 Emoji，不写大段英文说明
- 多仓库改动通常分别在各自仓库独立提交，而不是混成一次根级提交

## 提交标题规则

### 基本格式

```text
type: 中文描述
```

示例：

- `feat: 完善小程序资料和订单管理`
- `fix: 对齐资料管理与后端契约`
- `refactor: 使用 Poetry 管理部署虚拟环境`
- `chore: 优化 gitignore 配置`

### 强制要求

- 必须使用 Conventional Commits 前缀
- 描述默认使用中文
- 严禁使用 Emoji
- 不要写成纯英文标题，除非用户明确要求
- 标题应直接描述改动结果，不写模糊表述如“update code”“fix issue”

## 类型使用规则

- `feat`
  - 新功能、新页面、新接口、新业务能力
- `fix`
  - Bug 修复、字段对齐、兼容修复、流程修复
- `refactor`
  - 重构、结构优化、实现方式调整，但不改变主要业务能力
- `chore`
  - 配置、忽略规则、脚本维护、工程杂项
- `style`
  - 少量纯样式、排版、格式层调整；若改动同时包含行为变化，优先使用 `feat` 或 `fix`
- `docs`
  - 少量文档更新；仅在确实只是文档改动时使用

若没有明确理由，不随意扩展过多提交类型，优先保持与现有历史主流风格一致。

## 粒度规则

- 一次提交优先只做一类明确变更
- 不把无关文件、临时文件、顺手格式化、调试输出混进同一提交
- 若 backend / admin / minigram 同时改动，应分别进入各自仓库提交
- 文档同步若紧跟某次功能改动，可与该改动同仓库同主题提交；若是独立规则整理，应单独提交

## 变更说明规则

- 标题优先写“结果”，而不是过程
- 若是对齐类修改，优先使用“对齐”“同步”“修复”“完善”这类词
- 若是部署和工程调整，优先使用“优化”“重构”“完善部署脚本”这类词
- 若是纯规则或忽略项调整，优先使用 `chore`

## 历史改写规则

- 已推送的 commit message 如需修改，必须明确意识到这是 **改写历史**
- 改写后推送远端应使用 `--force-with-lease`
- 改写 message 时只改消息，不改提交内容
- 改写前若工作区有未提交修改，先暂存再恢复，避免污染历史操作

## 禁止事项

- 不要在根目录这个非 Git 仓库位置执行提交
- 不要使用纯英文且缺少上下文的信息，如：
  - `fix bug`
  - `update`
  - `refactor code`
- 不要把临时脚本、导出文件、调试结果直接提交
- 不要把多个子项目的无关修改揉成同一个提交
